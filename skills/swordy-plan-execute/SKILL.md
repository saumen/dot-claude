# Plan Execution Skill

Reads a structured plan document, executes its checklist items in order using task tracking APIs, syncs status to the on-disk plan file, and reports final status.

## Agent Routing

- **Must be executed by spawning the execute sub-agent using the Agent tool with subagent_type: swordy-agent-execute. Do not execute inline.**
- **Fallback:** If the execute agent fails twice (stream disconnect or other error), run the workflow inline using this skill's steps. Do not retry the agent.

## Workflow

### 1. Locate the plan document
- The plan is a markdown file at `docs/plans/{type}/{timestamp}__{slug}/plan.md` (e.g., `docs/plans/features/2026052400__add-oauth-login/plan.md`)
- If the user provides a path, use it directly. Otherwise, look for the most recent plan in the relevant subdirectory.
- Verify the file exists and contains a checklist section.

### 2. Parse the checklist
- Extract all checkbox items: `- [ ] **[TASK-XX]** ...`, `- [ ] **[TEST-XX]** ...`, `- [ ] **[VERIFY-XX]** ...`
- Group them by category (Production Code, Tests, Documentation, Verification).
- Note any implicit ordering (milestones, dependency notes).

### 3. Create the task list via task tracking APIs
- Call TaskCreate for each checklist item to create individual tasks.
- Set all tasks to `pending` initially.
- Include a brief explanation summarizing the plan's scope.

### 4. Execute tasks in order
- Work through steps sequentially, respecting any documented dependencies.
- For each step:
  1. Read the step description to understand what needs to be done.
  2. Perform the work (edit files, run commands, write tests, etc.).
  3. After completing the step, call TaskUpdate to set that step's status to `completed`.
  4. **Sync to disk**: Update the corresponding checkbox in the plan file on disk from `- [ ]` to `- [x]`.
- Run the test suite or verification commands after each milestone if the plan specifies it.

### 5. Handle failures
- If a step fails, stop and report the failure with context.
- Do not proceed to subsequent steps until the failure is resolved.
- Update the failed step's status via TaskUpdate (not `completed`) in both task tracking and the on-disk plan file.

### 6. Final verification
- After all tasks are completed, run the plan's verification steps (e.g., full test suite, integration tests).
- Call TaskUpdate for each remaining task to mark all steps `completed` with a summary explanation.
- Ensure the on-disk plan file has all checkboxes marked `- [x]`.

## Parallelization Guidance

Not Applicable. Plan execution is a sequential process.

## Scope

Not Applicable. The skill's frontmatter description defines when to use it.

## Key Rules
- **Always use task tracking APIs** — never rely on markdown checkboxes alone for tracking.
- **Sync to disk** — after every status change, also update the corresponding checkbox in the plan file on disk (`- [ ]` → `- [x]`).
- **One step at a time** — mark each step `completed` only after the work is done.
- **Respect dependencies** — do not skip steps or reorder without explicit user approval.
- **Report failures immediately** — do not hide errors or continue past broken steps.
